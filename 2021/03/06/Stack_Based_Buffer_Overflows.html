Stack Based Buffer Overflows

Hey everyone. Today we’ll be looking at stack based buffer overflows, which is an attack technique that can be utilized against some vulnerable programs, if they’ve been written poorly. This style of attack comes under the category of Binary Exploitation, because, as you might have guessed, we’ll be exploiting a compiled binary in order to get what we want.

This will (hopefully) be an in-depth introduction to the topic, with plenty of pictures and examples to guide you along. We’ll be using ROP Emporium’s excellent “ret2win” challenge here, which is the most basic of binary exploits. Let’s begin.

A peek at what we’ll be working with later on!

Section 1: Learning the basics

In this section, I’ll cover some of the basic terminology associated with Binary Exploitation, along with some of the different components that we’ll come across whilst solving these kinds of challenges. This will include things such as the basic assembly instructions that make up a compiled program, CPU Registers, and more.

1.1: What is binary exploitation?

Binary Exploitation is a type of exploit that has the attacker leverage a compiled program into doing something it’s not supposed to do. An example of this would be taking a program and using it to gain command execution on a system. Binary exploitation takes a vulnerability in a program, and turns that into a vector for making the program do things it’s not supposed to do. There are plenty of ways that this can happen, but today we’ll be keeping it as simple as it gets, focusing only on Stack-Based exploitation of a program. We’ll get into that in a moment.

1.2: How do these kinds of attacks work?

When a program is ran by a user, the program is loaded into memory (the RAM on your computer), and it’s here that all of the instructions take place. The program will have been compiled, meaning that the source code (the bit that we write) has been converted into machine code (the bit that the computer can understand). Once the program is loaded into memory, it executes a sequence of instructions, in the order that the compiler thinks is most efficient, such that the computer does what the programmer wanted.

Here’s what some instructions could look like. These would execute sequentially, from top to bottom.

Once the instructions are running, it may be possible for a user to hijack these instructions, making the program jump around in memory, into places it’s not meant to go. More on this in a short while. If a user can force the program to act in a specific way, they could eventually force it to execute commands for them, or even spawn an interactive shell.

1.3: What is the stack?

The idea behind this article is to talk about Stack-Based Buffer Overflows, which means we need to talk about what the stack actually is. The stack is a data structure in memory, which used for temporarily storing values for later use. You can either push items onto the stack, or pop items off of the stack. The stack is a Last-In-First-Out (LIFO) data structure, meaning that the latest item to be pushed onto the stack will be the first item to be popped off of it. A prime example of what might be stored on the stack is user input. If a buffer is allocated in the program, for example in C using char buf[64], then 64 bytes of space will be allocated to the stack, ready to be filled with user input. Items like local variables are also stored on the stack. You should also note that other important values, such as the return address are also stored on the stack. This will become important later.

Here’s a view of the stack just after some user input has been taken. The Buffer that the input is read into occupies a certain amount of the stack, so this is where the user input (The A’s) end up being stored. It looks very confusing, but much of the stuff on the stack is what we’ll refer to as “junk”. It’s unimportant for what we want to do.

1.4: Memory Addresses and CPU Registers

When a program is loaded into the memory, it is assigned a Virtual Address Space, which is a private instance inside of the memory which only this program can access. This is granted by the Kernel, and the areas inside this space are mapped out and given a specific address. This is to stop a program from accidentally breaking other programs, which are also stored in the memory. Once the program is over, the allocated address space is given back to the pool of available space, allowing the Kernel to re-allocate it to another program if it needs to.
